1. System Overview

Ogun Bridge is a centralized identity management and authorization hub. It mediates authentication and authorization between Authentik (the SSO/IdP), 389 Directory Server (LDAP user store), and consuming applications (Spectres Pantheon, Groove Payroll, Thoth ESU Gateway).

3. Authorization Resolution Chain

The core resolveRole() function (authorizer.js:4-158) resolves a user's role for a given app. Resolution priority:

Step 1App lookup by slug→MISSING→ { authorized: false }
Step 2Manual override check (is_override)→MATCH→ return override role
app_users.is_override = true → skip all group-mapping
Step 3Pre-resolved role from Authentik→MATCH→ look up by name, return
Only 'admin' | 'password_manager' | 'viewer' accepted
⚠ Used ONLY for Ogun's own OIDC login (ogunClaim.role)
Step 4Access group membership check→NOT MEMBER→ { authorized: false }
Must be in app's Authentik access_group to proceed
Step 5Group → role mapping resolution→MATCH→ highest priority wins
Queries group_role_mappings ordered by priority DESC
First case-insensitive match wins
Step 6Default role fallback→ALWAYS→ { authorized: true }
is_default role or viewer base role
⚠ System NEVER denies a user — worst case is viewer

Consumer App Auth Flow

  1. User logs into consumer app via the app's own Authentik OIDC client
  2. Consumer app's "onAuthorize" hook fires with { sub, email, groups }
  3. Consumer app calls POST /api/authorize with its static X-Api-Key
  4. Ogun Bridge validates the API key, calls resolveRole()
  5. Returns { authorized, roleDefinition, permissions }
  6. Consumer app stores result in its own session
  7. No push mechanism: Role changes are not propagated to consumer apps. They only take effect when the consumer app's session expires.

7. Ogun Bridge's Own RBAC

Ogun Bridge uses Authentik as its RBAC authority — not its own group_role_mappings table. The custom Authentik scope "Ogun Bridge Scope" runs a Python expression on every authentication request:

all_groups = [g.name.lower() for g in request.user.all_groups()]
group_set = set(all_groups)
if 'ogun-bridge' not in group_set:
    return {'ogun': {'hasAccess': False, 'role': 'none', 'groups': list(group_set)}}
role = 'viewer'
if 'systems_admins' in group_set:
    role = 'admin'
elif 'password_manager' in group_set:
    role = 'password_manager'
return {'ogun': {'hasAccess': True, 'role': role, 'groups': list(group_set)}}

This means:

Authentik GroupRole AssignedRBAC Capabilities
systems_admins (in ogun-bridge)adminFull access: users R/W, settings R/W, password manage
password_manager (in ogun-bridge)password_managerPassword R/W/manage, users R-only, dashboard R-only
In ogun-bridge onlyviewerRead-only: dashboard, logs, audit
Not in ogun-bridgedeniedNo access (hasAccess: false)
DB group_role_mappings exist as fallback but are unused during OIDC login
authorizer.js:55-68
The ogun.role claim short-circuits resolution at step 3, skipping the DB mappings entirely. Three mapping records for ogun exist in the DB (systems_admins→admin, password_manager→password_mgr, ogun-bridge→viewer) but they only activate if Authentik stops sending the ogun claim.

8. Gaps & Weaknesses

HIGH Severity

HIGH
H1 6 RBAC endpoints have no authentication
rbac.js:119,167,377,665,842,854
Any unauthenticated request can enumerate all apps, role definitions, group→role mappings, module schemas, Authentik groups, and base roles. This exposes the entire RBAC structure to anyone who can reach port 3333.
HIGH
H2 ID token never cryptographically verified
auth.js:108
The id_token JWT is received from Authentik but never validated. No signature verification, no aud/iss/exp checks. While HTTPS protects transport, a compromised Authentik or TLS failure could forge identities. The system trusts the userinfo response entirely.
HIGH
H3 API keys stored in plaintext
db.js:305
Consumer app API keys are stored as plaintext VARCHAR in the apps table. No hashing, no rotation mechanism, no key expiry. A DB breach exposes all service-to-service authentication credentials.
HIGH
H4 Consumer apps self-report user groups
authorize.js:13
The groups array is passed by the consumer app in the request body. Ogun Bridge has no independent verification that the user actually belongs to those Authentik groups. A compromised consumer app could inflate a user's privileges.
HIGH
H5 No role change push mechanism
When an admin changes a user's role (override or group mapping change), there is no webhook, pub/sub, or event system to notify consumer apps. Role changes only take effect when the consumer app's session expires or the user re-authenticates. This creates a window where a user retains outdated permissions.

MEDIUM Severity

MED
M1 Default fallback always returns authorized: true
authorizer.js:132-138
If no group mapping matches and no default role exists, the system falls back to the viewer base role. An unknown user with no group memberships still gets viewer access to every app. The system never truly denies access.
MED
M2 preResolvedRole accepts only 3 hardcoded values
authorizer.js:55
Only 'admin', 'password_manager', and 'viewer' are accepted as pre-resolved roles. If Authentik sends a different role name (e.g., a custom per-app role), it's silently ignored and falls through to group-mapping resolution.
MED
M3 syncUsersForApp makes N+1 API calls to Authentik
authorizer.js:270-340
The sync function fetches all group members, then for each member makes a separate API call to fetch their groups. No pagination handling. For large apps (e.g., 100+ users), this creates 100+ sequential API calls.
MED
M4 authenticate uses .then() not await
auth.js:60-89
The authenticate middleware wraps an async operation in a synchronous function. If an exception is thrown before the .then() executes, it falls outside Express's error handler and crashes the request.
MED
M5 protectPasswordOperation has race condition
auth.js:206-228
The function calls getUserOgunRole via .then() but calls next() before it resolves. The response can be sent before the async access check completes. For password_manager users targeting admins, the check might not execute.
MED
M6 No audit logging for denied access
auth.js:101,119,138
requireRole, requireSuperAdmin, and requireModule all return 403 with generic messages but write nothing to the audit log. There is no way to investigate who tried to access what and was denied.
MED
M7 claim_name column is dead — seeded but never read
db.js:307
The claim_name column on the apps table is configured for all 4 apps in seed data (ogun_role, spectre_role, groove_role, thoth_role) but is never referenced anywhere in the application code. This was intended for per-app Authentik OIDC custom claim resolution but was never implemented.

LOW Severity

LOW
L1 Dual role tracking: auth_users.role vs roleDefinition.name
auth.js:159
New OIDC users are created with auth_users.role = 'member', but RBAC middleware uses req.user.roleDefinition.name. The member value is never checked by any authorization gate.
LOW
L2 No session token rotation
auth.js:177
Session tokens are static 128-char hex values valid for 7 days. No refresh mechanism or token rotation on use. A leaked token remains valid for its full lifespan.
LOW
L3 Schema discovery depends on Docker internal hostnames
db.js:752-756
Seed schema endpoints point to Docker hostnames (groove:5005, spectres:8764, api:3010). These may not resolve from the Ogun Bridge container if the networks aren't configured correctly, causing silent auto-discovery failures.
LOW
L4 No per-app rate limiting on /api/authorize
Only global rate limiting (100 req/min) applies. A misbehaving or compromised consumer app could exhaust the global rate limit, affecting all apps and the UI.

9. Recommended Fixes (Priority Order)

PriorityFixAddressesEffort
P1 Add optionalAuth or requireAuth to all 6 unprotected RBAC read endpoints (GET /apps, GET /schema/:slug, GET /roles/:slug, GET /mappings/:slug, GET /authentik-groups, GET /base-roles) H1 Small
P2 Implement JWT ID token validation — verify aud (client_id), iss (issuer), exp (expiration), and RS256 signature using Authentik's JWKS endpoint H2 Medium
P3 Hash API keys at rest (bcrypt or SHA-256 with per-key salt), add rotation endpoint, add key metadata (created, last_used, expires_at) H3 Medium
P4 Add webhook mechanism: configurable URL per app that Ogun calls on role change events (override updated, mapping changed, sync completed) H5 Large
P5 Add audit logging to all 4 authorization middleware denial paths (requireRole, requireSuperAdmin, requireModule, requireLDAPGroup) M6 Small
P6 Add Authentik group verification in POST /api/authorize — fetch user's groups from Authentik API and compare against the groups passed by the consumer app H4 Medium
P7 Implement pagination + batch group fetch in syncUsersForApp. Fetch all group→user→group relationships in bulk instead of N+1 sequential calls. M3 Medium
P8 Remove or implement the claim_name column on apps table. Either wire it into the Authentik scope resolution or drop it. M7 Small
P9 Convert .then() patterns to async/await in authenticate and protectPasswordOperation middleware M4, M5 Small
P10 Remove the auth_users.role = 'member' dead branch or align it with RBAC roleDefinition L1 Trivial
